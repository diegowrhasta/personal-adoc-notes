**Backups and restoration**

[NOTE]
====
First of all, if you are going to restore a backup, you need to copy it to the container
itself, since that's where the service will look to restore.

Example:
`sudo docker cp /home/diego/Documents/projects/AdventureWorks2019.bak sqlserver2019:/var/opt/mssql/backup/`
====

After that you can then run a backup restore in the form of a query:

[source, sql]
----
USE master
GO <.>

RESTORE DATABASE [AdventureWorks2025]
FROM DISK = N'/var/opt/mssql/backup/AdventureWorks2019.bak' <.>
WITH 
    MOVE N'AdventureWorks2019' 
        TO N'/var/opt/mssql/data/AdventureWorks2019.mdf', <.>
    MOVE N'AdventureWorks2019_log' 
        TO N'/var/opt/mssql/data/AdventureWorks2019_log.ldf', <.>
    REPLACE,
    RECOVERY,
    STATS = 10
GO
----
<.> DBeaver doesn't recognize GO as a delimiter, it's multi-vendor purpose and
doesn't adhere more than a generic SQL syntax highlighter. You can configure it
to cater to that specifically, but don't know if that's really worth it
<.> As you can see, this is the path to which we copied previously from the host machine
<.> And then we have to move both the database file to a path the service understands,
a simple restore tries to use windows conventions like "C:\" and the container which
is Linux will not work at all, we have to manually tell it the path
<.> Same for the log file

After running the query, we should have successfully restored the `AdventureWorks`
database and we should be able to start running queries on it.

_And of course, we can start generating the diagrams that DBeaver does amazingly
well_

== Scoping your SQL Project

- We want to build a sales dashboard for AdventureWorks. How should you start?
A great starting point is to determine the scope of the project. Which is:

Create a list of questions you want to answer with your data:

- What are the total monthly sales?
- What are the monthly sales by country?
- Which products are our top sellers?
- Which stores perform the best?
- What is the average size of each order?
- What is the average customer lifetime value in each country?

As we dive into the data and write our queries, **we may refine these questions and
adjust our queries accordingly**

This is how you should always start your SQL projects: write a list of questions
you have for the data and then write the queries that will give you the answers.
_As you work through the queries and data, new questions will come up. Try to answer
them with your queries too_

=== Report 1: Monthly Sales

**Question: What is the company's revenue for each month**

The course uses syntax that's not `T-SQL`, like, wtf.

[source, sql]
----
SELECT
	YEAR(soh.OrderDate) AS OrderYear, <.>
	MONTH(soh.OrderDate) AS OrderMonth <.>
FROM Sales.SalesOrderHeader soh 
GROUP BY
	YEAR(soh.OrderDate), <.>
	MONTH(soh.OrderDate) <.>
ORDER BY
	OrderYear DESC, <.>
	OrderMonth DESC
----
<.> We are using the `T-SQL` function to extract the year of a date column
<.> We are using the `T-SQL` function to extract the month of a date column
<.> We are also using the same conversion of year for grouping
<.> We are also using the same conversion of month for grouping

So, I know this rule from the course I did:

[IMPORTANT]
====
Avoid functions on columns in the WHERE clause (when filtering on indexed columns)
====

So, `WHERE YEAR(OrderDate) = 2014`. Is **bad for performance**, **_if `OrderDate`
is indexed_** because:

- SQL Server must compute `YEAR()` for every row
- The index cannot be used efficiently
- It often results in an index scan instead of an index seek

Better:

````
WHERE OrderDate >= '2014-01-01'
  AND OrderDate < '2015-01-01'
````
**This allows an index seek**

- Group By => HAPPENS AFTER FILTERING. So it's normal to use functions in these
statements
- Select => Functions are fine here also

**Functions are dangerous when they prevent index usage on:**

- WHERE
- JOIN
- ON
- HAVING

If you frequently group by year/month, a production-grade solution can be:

**A persisted Computed Column**

[source, sql]
----
ALTER TABLE Sales.SalesOrderHeader
ADD OrderYear AS YEAR(OrderDate) PERSISTED; <.>

CREATE INDEX IX_OrderYear ON Sales.SalesOrderHeader(OrderYear); <.>
----
<.> This is a computed column, but that is also saved to disk. **And automatically
is recomputed on update (and of course on insert)**
<.> This indexes the computed column so that queries are faster

But only do this if:

- You frequently filter/group by year
- Table is large
- Read-heavy workload

It has tradeoffs, of course, because of the extra computation, but they should be
minimal in the face of making reads faster. But it's something big, and you'd
be jumping the gun if you go all in from the start. Always look at the results,
at the demand first.

Okay, lastly we actually want an aggregation, cuz right now the query is just
imputting all the years and months lmao.

[source, sql]
----
SELECT
	YEAR(soh.OrderDate) AS OrderYear,
	MONTH(soh.OrderDate) AS OrderMonth,
	ROUND(SUM(soh.TotalDue), 2) AS TotalRevenue <.>
FROM Sales.SalesOrderHeader soh 
GROUP BY
	YEAR(soh.OrderDate),
	MONTH(soh.OrderDate)
ORDER BY
	OrderYear DESC,
	OrderMonth DESC
----
<.> And so, after we have grouped by the two columns, we are going to aggregate
the `TotalDue` amount with `SUM` and then have the result rounded to **two
decimal points**

**NOTE:** This is literally a query on **ONE TABLE**, so it's really simple
if you look at the bigger picture, the biggest "take-away" from here is that
we are leveraging groups, orders, and aggregator functions (on top of formatting
such as `ROUND`)

=== Report 2: Monthly Revenue by Country

**Question: What is the monthly revenue for each country?**

We would like to see the total revenue for each month (of each year) for every
country in the database. The result should be sorted in chronological order, with
the more recent dates coming first. We can use this data to plot trend lines for
each country or analyze the data as-is.

Okay, so this is interesting we are now going `cross-schema`. We were at the
`Sales` schema for Report 1, but now we will jump to the `Person` schema for
two tables that live there and are connected to our `SalesOrderHeader` and
`SalesTerritory`.

[source, sql]
----
SELECT
	cr.Name as Country, <.>
	YEAR(soh.OrderDate) as OrderYear,
	MONTH(soh.OrderDate ) as OrderMonth,
	ROUND(SUM(soh.TotalDue), 2) AS TotalRevenue
FROM Sales.SalesOrderHeader soh
JOIN Sales.SalesTerritory st <.>
	ON soh.TerritoryID = st.TerritoryID 
JOIN Person.CountryRegion cr
	ON st.CountryRegionCode = cr.CountryRegionCode 
GROUP BY
	cr.Name, <.>
	YEAR(soh.OrderDate),
	MONTH(soh.orderDate)
ORDER BY
	OrderYear DESC,
	OrderMonth DESC,
	Country <.>
----
<.> We are building on top of the previous query we made, just adding a new `Country`
dimension to it.
<.> But in order to get to this new dimension (table) we have to start joining with
other tables, and in here you see how we are passing from `Sales` to `Person` (all
tables have as a prefix the name of the schema they belong to)
<.> We are also adding the new country name dimension as a grouping parameter so that
we get the aggregation for the respective records
<.> And we also want the country to be alphabetically ordered

So this is really, really interesting. I like that we are building on top of the
previous query, adding extra concepts one by one, and introducing new ideas and
things along the way.

It's funny how sometimes we use the aliases and other times the fully qualified name,
we should remain as consistent as possible. Besides that I **really like** the
way we are formatting the SQL, it looks so neat.

_Note:_ There's a lot of data, about 228 rows are returned from the previous query

=== Report 3: Bestselling Products

**Question: What are Our Bestselling Products?**

We would like to add a list of the bestselling products together with some relevant
information like lifetime sales and customer reviews. The report should show
reasonably recent data.

The main metric in the query is the total quantity sold for each product, which
can be obtained by using the combination of `SUM(OrderQty)` and `GROUP By ProductId`.

**But here's the kicker,** just doing that, would display to us some `ProductId`s
records, which is just **hard to read**. We need more information if we want
to give to the public useful info.

_Note:_ And here comes another lesson on both database design and overall expectations
for data model structures: `Not everything will be connected logically`. But you
can always rely on the fact that your **NAMES** should help you bridge gaps. Another
great argument to keep naming conventions, and naming consistency.

As the result of the query only on `SalesOrderDetail` would be hard to read,
we can use its `ProductId` column to join to the `Product` table of the `Production`
schema. If we join the two tables, we can access each `ProductId`'s `ProductName`,
which leaves us this query:

[source, sql]
----
SELECT TOP (10) <.>
  p.ProductId,
  p.Name AS ProductName,
  SUM(od.OrderQty) AS TotalUnitsSold <.>
FROM Sales.SalesOrderDetail od
JOIN Production.Product p
  ON od.ProductID = p.ProductID <.>
GROUP BY
  p.Name,
  p.ProductId
ORDER BY TotalUnitsSold DESC
----
<.> The website we are using to study uses `LIMIT 10` at the end, but that's not
valid in MSSQL, for getting only a subset we use (generally) `TOP`
<.> An easy alias application for the aggregation of the quantity of ordered
items
<.> Here comes the joining on the `ProductID` column, notice the exact same name,
which gives us a clue into the fact that these two columns hold the same values.

_Note:_ Besides naming conventions being good contextual clues, it's worth
pointing out that this is a possible risk/issue if a developer just doesn't know
about the convention. Worse if your names have no relation at all, it turns into
you just "having to know somehow". And memory is fragile. It always is.

==== Foreign Key Constraints

For a quick TL;DR:

- Use foreign keys by default
- Disable only when you have a specific performance reason

_Most systems are harmed more by missing constraints than by having them_

But, if we want to go a bit deeper:

A `JOIN` only requires matching values - not a constraint.

So you might ask:

> If joins work anyway, what do FK constraints actually give you?

In Microsoft SQL Server, the advantages are significant.

**Reason 1:**

Without a foreign key:

````
Orders.CustomerID = 9999
````

Even if customer 9999 does not exist, SQL Server will happily allow this.

With a FK constraint:

````
ALTER TABLE Orders
ADD CONSTRAINT FK_Orders_Customers
FOREIGN KEY (CustomerId)
REFERENCES Customers(CustomerID)
````

With this, SQL Server guarantees:

- You cannot insert an order for a non-existent customer
- You cannot delete a customer that still has orders (unless cascade is defined)

This prevents:

- Orphaned rows
- Silent data corruption
- Broken business logic

And that is no _small benefit_, it's structural safety

**Reason 2:**

Query Optimizer Improvements. The SQL Server's query optimizer can use FK relationships
to:

- Eliminate unnecessary joins
- Improve cardinality estimation
- Generate better execution plans

Example:

If a FK guarantees that every `Order.CustomerId` exists in `Customers`, SQL Server
knows:

- The join cannot multiply rows
- The relationship is 1-to-many
- Certain joins may be simplified

Without the FK, the optimizer must assume less.

So yes - performance can improve

**Reason 3:**

Self-Documenting Schema

With FK constraints, someone inspecting the database instantly understands:

- Which tables relate
- How they relate
- Cardinality direction

Without FKs, relationships exist only in the developer knowledge. **In large systems,
that becomes dangerous**

**Reason 4:**

Cascading Rules (Optional but Powerful)

You can define:

`````
ON DELETE CASCADE
ON UPDATE CASCADE
`````

This makes the database enforce relational behavior automatically

Without FKs you'd need:

- Triggers
- Application logic
- Manual cleanup

Which increases bug risk.

So yeah, you can trigger a delete and that can cascade across all related tables
that have a constraint connecting them (making your life easier if you are just trying
to tear down everything)

**Downsides?**

There's always a tradeoff:

- Slightly Slower Writes: On INSERT/UPDATE/DELETE, SQL Server must check referential
integrity
- Bulk Loads can be Slower: In data warehouse staging, people sometimes disable FKs
for performance
- Migration Order Matters: You must insert parent rows before child rows

So, _when do people avoid FKs?_

Some teams skip FKs in:

- High-throughput logging tables
- Data warehouse staging
- Distributed microservices
- Systems where integrity is enforced at application layer

But in OLTP systems? - FKs are almost always recommended

=== Back on track with Mission 3

With a ProductId, a ProductName and the total number of units sold, the result
data set is already a viable dashboard element. But we can go _further and
beyond_.

So our first part of the query already gives us a lot of leeway, we now have a
foundation to work with products, _we have effectively established the bridge
to go from `Sales` to `Production`_.

We want to also add the product ratings, for an even deeper insight into our
data.

**IMPORTANT:** Do not try to build everything on the big query, you can divide
and conquer, so we can start on the other side of the spectrum, and start shaping
the data we want extracted from there (write a simple separate query first).

So, when it comes to product ratings, we have the `ProductReview` table of the
`Production` schema. In there we care about `ProductId` and `Rating` left by
the customer. To get a better insight, we'll average the rating scores and group
the data by `ProductId`

[source, sql]
----
SELECT
  ProductId,
  ROUND(AVG(Rating), 1) AS ProductRating
FROM Production.ProductReview
GROUP BY ProductId;
----
Pretty easy to get, right? Just by looking at the result we can also infer even
more. Not many people write reviews, we literally have only 3 products with reviews
and data to work with. Which is also key information for the building of the bigger
query that will join this rating data with what we had with total number of
units sold.

We will now learn about a new concept that's really cool, and helps us in not
making ugly joins/where statements that do full table searches because we are using
IN or subqueries (not performant).

And that's: `Common Table Expressions` (CTE).

There's a guide:

https://learnsql.com/blog/what-is-common-table-expression/

But as cliff notes here are some great advantages they provide:

- More readable queries
- Manage format differences between table data and report data
- Avoid bad-performing sql by simply computing data separately into a "virtual
table", that we can then work with
