== Goodbye Microservices, Hello Self-Contained Systems - Simon Martinelli

Notes taken from https://www.youtube.com/watch?v=hW22FxcrtkA

2010 => Netflix pushed for microservices

They created a **big ball of mud**. Mainly because they didn't understand microservice
architecture well.

**Misconception**

- Monolithic Applications = Big Ball of Mud

It could be a Big Ball of Mud, or it could be glorious. You can build _Modular
Monoliths_ in a modular way. Java does a bit of that, _but it might not be a
good idea_.

A good idea to keep in mind:

[quote, David L. Parnas, On the criteria to be used in decomposing systems into modules]
----
The **effectiveness of a modularization** is dependent upon the **criteria used in
dividing** the system into modules
----

If you have a lot of coupling within the modules, then things might be useless.

It's all about Modularity.

**Module**

Modules can be in microservices, in packages, in components. They all share the
same definitions:

A self-contained **component** of a system, often **interchangeable**, which has
a **well-defined interface** to the other components.

But let's move now to **Microservices**

- Naming is hard, however using **microservices** to talk about modules that can be
an error.
	- Micro is not micro, it just means "smaller"
	- Domain Driven Design => Bounded Contexts was a good concept to borrow. It
	is a framework to split a big domain into subdomains. You can separate a
	business by some form. _The organization of the company is a good starting
	point to see how you can define your bounded contexts_

_One issue with MS:_

Some customers have so many microservices, that they can't even update easily
to the newest technologies and they realize late that they could've just put them
all together for simplicity.

**Interesting idea**

- Even if your app is a Big Ball of Mud, you can use tools, and nowadays AI
to re-engineer your application so it's more modular, separating concerns
better and making it more maintainable and extendable.

Now, if you have _a distributed big ball of mud_. Then that's much more complicated.
There's people that are doing distributed tracing in production just to find out
what microservice calls to what other microservices.

[quote, Simon Brown]
----
If you can't build a modular monolith, what makes you think microservices are the
answer?
----

In separating things into separate services, you could even make an application
_slower_.

[quote, Martin Fowler]
----
First Law of Distribution: Don't Distribute
----

By distributing things you might end up with:

- Higher latency
- Security concerns
- Problems with resilience
- Network might not be fast/reliable

If you do communication between microservices:

- Synchronous => REST services
	- Netflix doesn't use REST anymore, they do GraphQL for Front, gRPC for backend
- Asynchronous => Helps with resilience (don't wait for response)
	- You send the request, and at some point you get the reply
	- But what do you do if you wait for it and don't get an answer?
		- Send request again
			- If the consumer is not idempotent, things are done twice
- Event based => Fire and forget
	- Things get consumed at some point, and perhaps further events are created
	based on the process of the previous ones
	- But apps become more resilitient as a consequence
	- **But there's a price**, you need some messaging system. You need something
	like RabbitMQ, Apache Kafka. This has to be maintained, upgraded. So _NOT FOR
	FREE_

=== Anti-Patterns

Integration Database = Multiple Microservices are accessing the same schema on
the database, and you have issues of: Who does migrations? Who's the owner of
the data? Who's writing the data? - So please, don't do it.

However, the **Integration Database** can become an intermediate step from a monolith
to a more modular system.

Moving the backend to modularity and microservices might solve different issues,
however the bottleneck can easily drift into the Frontend. The Frontend might
become this huge monolith too.

_Note:_ So companies are moving to micro-frontends, but apparently at the time
of writing (2026-June) there really aren't mature frontends to really take on
the whole micro-frontends pattern in stride.

=== Self-Contained Systems Architecture

Apparently, this pattern was introduced at the same time that Netflix did the
microservices inception.

SCS:

- Cut the monolith along bounded contexts
- Wrap each domain in its own web application
- That application is a **Self-contained System (SCS)**

We don't create just a backend service, we create the whole application and that's
called Self Contained Systems (SCS)

**German speaking side:** Seems to know well about this.

_Characteristics:_

- An SCS contains its own **user interface**, specific **business logic**, and **separate
data storage**
	- It may contain an _optional API_ (preferably async)
	- This becomes even bigger than a microservice, and in turn, the communication
	across "micro-services" ends up being less important

So, you have the whole app in **one silo**.

**Data Storage**

- Every SCS brings its **own data storage**, which may cause **redundant data**
- Redundancy is tolerable as long as **data sovereignty** is preserved
	- So just one system should write, read

**How to Integrate SCS?**

- Self-contained Systems should be integrated via their **web interfaces** to
minimize coupling to other systems.
	- Basically, you are going to go back to good 'ol **hyperlinks**
		- You click on one place, and that navigates away to another SCS

You might be thinking: _Ah, that's so bad, my users want seamless interactions and
great integration_. But if you put some telemetry in place, you might be surprised
the amount of people that will stay within **one silo**

=== Asynchronous for the Win!

- **Synchronous** remote calls inside business logic should be avoided
- **Asynchronous** calls **reduce dependencies** and prevent error cascades
- **Consistency** guarantees **may be relaxed**, depending on the required update
rate

- So eventual consistency will be introduced
	
The advantages of SCD => Are the sames as microservices

- Resilient
- Scalable
- Open
- Independent

The most important part of modular architecture styles is the modularity.

Modernization => Monoliths to turn into modules.

- Scaling to Zero => An interesting strategy to save up on costs, is the fact that
on weekends when people don't use the systems you can scale back everything and
save up on some power
	- Making things modular allows for you to choose whatever tech stack you want,
	and also plug-and-play solutions if you choose so

**AI Needs Context**

- AI assistants are only as good as the context they see

**With microservices**

- One feature lives in many repos
- The full code may not fit in the context window
- AI suggestions break other services
- Humans still stitch it together

In this age of AI and Agentic coding, if a context is **too big** or **too chaotic**.
The AI itself might have a hard time really understanding it, and in turn waste
more time (even though they are amazing at reading and processing tons of information
and fast).

**SCS: Right-Sized for AI**

- **The full picture fits**
	- UI, business logic, and data access live together
	- The whole context fits in the window

**Changes stay inside the boundary**

- One team per SCS maps to one AI agent per SCS

So, you can program, maintain, understand, and iterate over an end-to-end
architecture, which brings in advantages such as less fragmentation, more locality,
less cognitive overload.

=== Modernizing a Wholesale ERP System

This is a DEMO of how a customer project by using SCS works and how it's been
developed.

An old homehrown ERP System. People don't want to use applications that must
be installed, UI is outdated, so a need for modernization was needed.

The domains that were done:

- Procurement
- Sales
- Logistics
- Finance

All the parts are there, logically. The first step:

- Split the monolithic application into the **high level subdomain struct**

Apparently they have about 30 SCS running to serve this modernized iteration.

And they leverage "DDD" because they are using "Bounded Context" to describe
how each SCS makes usage of their data to do their assigned work.

Everything is **events**, that get published to a topic/queue, and that then
get passed around amongst domains. They then take the data inside the events,
and update their databases accordingly, on top of reacting however they need
to react. Another big thing is that separated to these business logic events,
there's solely **database events**, replication. That happens amongst the
multiple databases, Even if you have read-only or write-only databases, you need
to replicate some level of data to keep consistency, and in case you need (data
RIGHT NOW), you would need to go through the topic/queue intermediary, and perhaps
have a bigger delay than what a sync endpoint call would do.

**Macro Architecture**

- **Data integration**
	- **Replication** of read-only data based on database views
	- Async through **events**
- **User Interface**
	- Common **Design Systems** with Components and Styles
	- **Link database** for navigation

SCS are **completely independent systems**. One can be written on JS, other on
Go, other on C#, other in JAVA. However, the decisions that was taken was as
follows:

````
- Because we want to exchange developers from different teams, all the application's
stack will be with the same framework/language.
````

In this specific use case, the different SCS were actually being used by different
users, and different departments, so they wouldn't really start navigating across
SCSs, **however**, the team still decided to run with a **Design System** so
all of them share a visual line.

**Data Redundancy**

There are two ways:

- You do **Replication**
	- **Pull-based** replication
	- **Target system refreshes data**
	- Good for **data** that **does not change** very **often**
	- Adds **delay** depending on refresh interval
	- **Simple** to implement
- You do **Events**
	- **Push-based** replication
	- **Source system publishes changes**
	- Good for **frequently changing data**
	- **Near-real-time** updates
	- **Higher** operational **complexity**
		- Someone has to manage the messaging system, the Kafla Cluster, etc

So, an interesting strategy for migration is as follows:

````
In moving the old system to the new SCS architecture. We want to keep running the
old systems as long as possible (yet, knowing that they will be gone eventually),
so that if something goes wrong with the new system. Users can fallback to the old
system.
````

The idea is to connect an old database with the new system (at least for the time
being), since the team sometimes doesn't know how many SCS will be there, it really
depends as you go, how you analyze, what you encounter, the issues and challenges
that start showing up the more you dig further. But the idea, always, is that the
old database will be gone as well eventually.

_Migration of a system sometimes takes longer than you envision_. Most of the time
is because you need the experts, the domain experts to chime in and help out.

_Management might think something will take 3 years_. But internally the team
might already have a rodemap for 5 years.

== On Designing an Observability Platform

There are tons of observability ecosystems, but one we can focus on because of
its popularity and easiness to pick up is: **The Grafana ecosystem**

- Grafana
- Grafana Loki
- Grafana Tempo
- Grafana Mimir
- OpenTelemetry Collector

Often referred to as the LGTM stack.

An observability platform can have different workload patterns competing for
resources:

. Metrics (high cardinality, steady writes, bursty reads)
. Logs (very high ingest + storage heavy)
. Traces (high fan-out, memory sensitive)
. SQL Server querying / dashboards (interactive, latency-sensitive)

And if you are going at full mega ultra scale, yeah, complexity grows to accomodate
for the thousands of needs. However if you are small, you can still benefit, and
actually is better if you shift left. That way you can grow way more easily.

Say you have about 100 transactions a day. Scale is nowhere near a problem.

This could be categorized as:

- "Foundational observability maturity" stage

At this stage you can build an **observability platform** correctly _before_
scale forces compromises.

The right goal for you is not:

- Hyperscale architecture
- Distributed telemetry clusters
- Aggresive sharding
- Massive retention engineering

The right goal is:

- Operational clarity
- Low maintenance
- Excelleng debugging visibility
- Future scalability without re-platforming

That leads to a very different recommendation.

**At this scale:**

- Traces are tiny
- Metrics are tiny
- Logs are manageable
- Query load is almost nonexistent

So the biggest risks are NOT:

- CPU
- Ingest throughput
- Disk IO

The biggest risks are:

- Operational complexity
- Poor telemetry design
- Overengineering
- Lack of standarization
- Observability nobody actually uses

_And this is extremely common in small/medium enterprise telemetry rollouts_

**The most valuable asset right now is NOT hardware_

It is:

**Telemetry consistency**

Meaning:

- Standarized trace names
- Good span attributes
- Correlation IDs
- Structured logs
- Service naming conventions
- Environment tagging
- Transaction identifiers

This is what makes observability powerful

Not 64 CPUs.

=== Most Critical things to take into consideration

**Invest in Trace Quality Early**

That matters MUCH more than infrastructure.

A good trace should answer:

- What happened?
- Where did latency occur?
- What dependency failed?
- Which SQL query failed?
- Which customer transaction broke?

Without opening multiple systems.

**What Good OTel Instrumentation Looks Like**

Include attributes like:

- service.name
- service.version
- deployment.environment
- transaction.id
- customer.id (carefully)
- sql.server.name
- db.operation
- http.route
- exception.type

But avoid

- High cardinality junk
- Raw query text as labels
- Session IDs everywhere

> We're small so observability can be informal

No, the opposite is true.

Smaller environments benefit massively from:

- Rapid troubleshooting
- Low MTTR
- Operational confidence
- Faster onboarding
- Reduced dependency on tribal knowledge

High cardinality junk:: Number of unique label/tag combinations. You shouldn't
label user ids, customer ids, they can be inside something that indexes them and
it has low cardinality
Low MTTR:: Mean Time To Resolution (how long it takes to fix production problems)

And let's introduce an OTel Collector Centralized Initially, that way, even if
we have to scale eventually, it will have to be simply adding a new gateway:

````
Services
   ↓
OTel Collector
   ↓
Loki / Tempo / Prometheus
````

- Centralized processing
- Easy config changes
- Batching
- Retries
- Normalization

And later you can split into:

- Agent collectors
- Gateway collectors

Without redesigning instrumentation.

=== Grafana Alloy

You don't have to build your own OpenTelemetry Collector, a "default answer" in
the Grafana ecosystem is: **Grafana Alloy**

It is a production-ready distribution of the OpenTelemetry Collector specifically
designed to integrate with hte LGTM stack.

- Runs beautifully in containers
- It is designed exactly for "small now, scale later"
- It avoids you having to hand-build a custom collector distribution

Be aware of 3 different meanings:

[cols="1,1"]
|===
|Meaning, Reality

|Build from source
|Not needed

|Create a custom distribution
|Not needed

|Write config pipelines
|Still needed minimally

|===

You WILL still configure:

- Receivers
- Processors
- Exporters

But you will NOT:

- Compile anything
- Maintain custom binaries
- Manage component compatibility

**The Industry Direction Right Now**

Historically:

- Grafana Agent
- OpenTelemetry Collector
- Prometheus agents

were fragmented

Grafana Labs clearly is consolidating around:

**Alloy as the unified telemetry pipeline**

Grafana introduced Alloy at GrafanaCON specifically as:

- "the OpenTelemetry Collector distribution with built-in Prometheus pipelines"

That matters because:

- Docs are improving rapidly
- Ecosystem support is growing
- Examples are plentiful
- Integrations are first-class

**One Alloy Container is enough for small**

````
Applications
   ↓ OTLP
Grafana Alloy
   ├── Loki
   ├── Tempo
   └── Prometheus
         ↓
      Grafana
````

_Note:_ Some advanced users say that Alloy might fall behind on some features
and whatever, but that's irrelevant for more use cases unless you need bleeding-edge
processors, hyperscale infra, uncommon exporters.

This would be a great start:

|===
|Container | Purpose

|alloy
|Collection pipeline

|Dashboards

|loki
|Logs

Alloy Features that matter operationally are:

1. Unified Telemetry = One collector, logs;metrics;traces. Instead of multiple agents
2. Built-in batching/retries = This matters during outages
3. Memory limiting = Critical protection against collector crashes
4. Native OTLP support = Applications can emit directly using OTLP/gRPC and OTLP/HTTP
5. Prometheus compatibility => Prometheus ecosystem remains massive, exporters are everywhere, infra metrics become easy later
